Scenario 102 — Genuine ES256 asymmetric JWT verification across two Workflow processes

App A
auth.m2m issuer
ES256 · /oauth/jwks
→ JWKS →
App B
sso.oidc verifier
jwksUri mode · /verify
1. Fetch a token from App A idle

Fetches an ES256 access_token from App A's /oauth/token (client_credentials grant).

2. Token to verify idle

The token is sent as Authorization: Bearer <token> to App B's /verify endpoint.

About this demo

App B holds only App A's public JWKS (fetched from http://app-a:8080/oauth/jwks via sso.oidc jwksUri mode). There is no shared secret between the two services. A token signed by any other key will be rejected, proving genuine asymmetric cross-service verification.
