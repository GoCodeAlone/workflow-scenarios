# Scenario Proof Quality Backlog

This backlog tracks scenarios whose tests do not yet meet the repository proof
standard. The audit was refreshed after scenarios 23-32 and 104-105 were
converted to real Workflow app/API proofs.

## Audit Method

Run:

```sh
scripts/audit-scenario-proof-quality.sh
```

This is a triage heuristic, not a final proof judgment. Scenarios can be valid
without HTTP if the scenario is explicitly a `wfctl`/CLI/tooling contract.

## Summary

- 102 scenarios have `test/run.sh` scripts.
- 63 scenarios currently exercise an API/application-style boundary.
- 89 scenarios exercise a Workflow host or `wfctl` boundary.
- 3 scenarios are package-test-only despite describing Workflow app or plugin
  behavior.
- 0 scenarios were classified as static-only without a Workflow/API marker.
- 3 scenarios have no first-pass proof marker and need manual classification.

## Highest Priority: Package-Test-Only Scenarios

These scenarios describe modules, plugin behavior, app endpoints, or lifecycle
flows, but their `test/run.sh` scripts only run Go package tests from the
`workflow` repo. They should be converted to real scenario apps or explicitly
reclassified as package-test support fixtures.

| Scenario | Current Test Shape | Expected Remediation |
|---|---|---|
| `33-apigateway-autoscaling` | `go test ./module -run TestAPIGateway|TestAutoscaling...` | Drive gateway/autoscaling plan/apply/status through the scenario config and assert state transitions. |
| `34-app-container` | `go test ./module -run TestAppContainer` | Drive deploy/status/rollback through the scenario config. |
| `35-multi-cloud-accounts` | `go test ./module -run TestCloudAccount` | Drive multi-cloud account validation through Workflow using mock/inline credentials. |

Already remediated: scenarios 23-32 now launch real Workflow app paths and
drive HTTP/API boundaries against local mocks or fixtures.

## CLI/Tooling Boundary Scenarios To Review

Many scenarios use `wfctl` without an HTTP/API boundary. Several are likely
valid because their subject is `wfctl` generation, validation, registry, secret
setup, or IaC planning. Review these for two things before changing them:

- The README/scenario description should claim a CLI/tooling contract, not an
  app behavior contract.
- Assertions should inspect output generated by the command under test, not
  static config strings unrelated to the command result.

Examples that deserve focused review after the package-test-only conversions:
`66-iac-multi-cloud`, `76-tofu-generate-pipeline`,
`84-enterprise-rag-pipeline`, `96-iac-dyndns-multiprovider`,
`98-ci-generate-smart`, and `100-ci-generate-jenkins-circleci`.

## Remediation Rules

- Prefer converting one scenario family per PR so failures are easy to isolate.
- Use the real Workflow host or `wfctl` command that owns the behavior.
- Mock dependencies at real seams only: in-memory stores, fake providers,
  local emulators, or committed fixtures.
- Keep scripts actor/resource-parametric where the proof vectors allow it.
  If a vector fixture is bound to a specific tuple, document that boundary.
- Update `scenarios.json` only from actual scenario runner output.
